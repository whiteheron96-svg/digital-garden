# GitHub Actions로 매일 아침 YouTube 뉴스 2개를 AI 요약해 Notion에 저장하기

매일 아침 반복해서 보는 YouTube 뉴스가 있다면, 새 영상을 자동으로 수집하고 AI로 요약한 뒤 Notion에 쌓아둘 수 있다. 이번 작업에서는 GitHub Actions, Gemini API, Notion API를 조합해서 무료 또는 저비용으로 운영 가능한 자동화 흐름을 만들었다.

목표는 단순했다.

- 매일 아침 정해진 시간에 실행
- 특정 YouTube 프로그램의 최신 영상만 수집
- AI로 한국어 요약 생성
- 원문 출처와 함께 Notion 데이터베이스에 저장
- 이미 저장한 영상은 중복 저장하지 않기

## 전체 구조

자동화 흐름은 아래와 같다.

```text
GitHub Actions 스케줄 실행
→ YouTube RSS 또는 보조 페이지에서 최신 영상 확인
→ 제목 키워드로 원하는 프로그램만 필터링
→ 공개 transcript 수집 시도
→ transcript가 없으면 Gemini의 YouTube URL 입력으로 요약
→ Notion DB에 원문 URL과 요약 저장
```

이번에 대상으로 삼은 프로그램은 두 개다.

- 한경 코리아마켓의 `모닝루틴`
- 한국경제TV의 `당신이 잠든 사이`, 즉 `당잠사`

`모닝루틴`은 YouTube RSS에서 잘 잡힌다. 반면 `당잠사`는 YouTube RSS 최신 목록에서 안정적으로 잡히지 않는 경우가 있어, 한국경제TV 웹페이지에 노출된 YouTube 링크를 보조 수집원으로 함께 사용했다.

## 왜 GitHub Actions를 썼나

로컬 Mac의 cron이나 launchd로도 충분히 만들 수 있지만, 노트북이 꺼져 있으면 실행되지 않는다. GitHub Actions를 쓰면 GitHub 서버에서 정해진 시간에 workflow가 실행된다.

이번 workflow는 하루 한 번만 실행되기 때문에 사용량도 크지 않다.

```text
매일 1회
Ubuntu runner
예상 실행 시간: 수십 초~몇 분
```

public repository라면 GitHub-hosted runner minutes 비용 부담이 거의 없고, private repository라도 이 정도 규모는 월간 무료 사용량 안에서 감당 가능한 수준이다.

## GitHub Actions yml 파일

GitHub Actions는 repository의 아래 경로에 있는 YAML 파일을 읽는다.

```text
.github/workflows/youtube-news-summary.yml
```

예시는 다음과 같다.

```yaml
name: YouTube News Summary

on:
  schedule:
    # 08:50 KST every day. GitHub Actions cron uses UTC.
    - cron: "50 23 * * *"
  workflow_dispatch:

jobs:
  summarize:
    runs-on: ubuntu-latest
    timeout-minutes: 20
    env:
      GEMINI_API_KEY: ${{ secrets.GEMINI_API_KEY }}
      NOTION_API_KEY: ${{ secrets.NOTION_API_KEY }}
      NOTION_YOUTUBE_DATA_SOURCE_ID: ${{ secrets.NOTION_YOUTUBE_DATA_SOURCE_ID }}
      NOTION_VERSION: "2026-03-11"
      YOUTUBE_SUMMARY_TIMEZONE: "Asia/Seoul"
      GEMINI_MODEL: "gemini-2.5-flash-lite"
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.13"
      - name: Install dependencies
        run: python3 -m pip install -r youtube_summary_bot/requirements.txt
      - name: Summarize YouTube videos to Notion
        run: python3 youtube_summary_bot/summarize_youtube_to_notion.py --max-videos 2 --max-age-hours 96
```

여기서 중요한 부분은 `cron`이다. GitHub Actions의 cron은 UTC 기준이다.

```text
50 23 * * *
```

이는 UTC 23:50이고, 한국시간으로는 매일 오전 08:50이다.

`workflow_dispatch`를 넣으면 GitHub Actions 화면에서 `Run workflow` 버튼으로 수동 실행할 수 있다. 처음 설정할 때는 반드시 수동 실행으로 테스트하는 것이 좋다.

## Secret은 GitHub Secrets에 저장

API 키를 코드에 직접 쓰면 안 된다. GitHub repository의 아래 메뉴에 secret으로 저장한다.

```text
Settings
→ Secrets and variables
→ Actions
→ New repository secret
```

필요한 secret 이름은 다음과 같다.

```text
GEMINI_API_KEY
NOTION_API_KEY
NOTION_YOUTUBE_DATA_SOURCE_ID
```

블로그나 공개 repository에 실제 값은 절대 올리지 않는다. 코드에서는 `${{ secrets.SECRET_NAME }}` 형식으로만 참조한다.

## Notion 데이터베이스 구조

Notion에는 요약 결과를 저장할 데이터베이스를 만든다.

속성은 아래 정도면 충분하다.

```text
Title: title
Program: select
Channel: rich text
YouTube URL: url
Published At: date
Processed At: date
Video ID: rich text
Status: select
```

중복 저장 방지는 `Video ID`로 한다. workflow가 다시 실행되어도 같은 영상 ID가 이미 Notion에 있으면 건너뛴다.

한 가지 놓치기 쉬운 부분이 있다. Notion API를 쓰려면 해당 데이터베이스를 Notion integration에 공유해야 한다. Notion DB 우측 상단 메뉴에서 `Connections` 또는 `Add connections`를 열고, 자동화에 사용할 integration을 추가해야 한다.

이 권한이 없으면 GitHub Actions에서는 이런 식의 에러가 난다.

```text
Could not find database with ID
Make sure the relevant pages and databases are shared with your integration
```

이 에러는 DB ID가 틀렸다는 뜻일 수도 있지만, 실제로는 integration 접근 권한을 안 준 경우가 많다.

## YouTube 수집 방식

YouTube 채널의 최신 영상은 RSS로 확인할 수 있다.

```text
https://www.youtube.com/feeds/videos.xml?channel_id=CHANNEL_ID
```

RSS에는 영상 제목, video ID, 업로드 시각 등이 들어 있다. 여기서 제목 키워드를 기준으로 원하는 프로그램만 골라낸다.

예를 들어:

```text
모닝루틴
당잠사
당신이 잠든 사이
```

다만 모든 프로그램이 RSS 최신 목록에서 안정적으로 잡히지는 않았다. 그래서 `당잠사`는 한국경제TV 페이지에 노출된 YouTube 링크를 읽고, YouTube oEmbed API로 제목을 확인하는 보조 경로를 추가했다.

핵심은 하나의 수집원이 실패해도 전체 workflow가 죽지 않도록 만드는 것이다.

## Transcript가 없을 때의 처리

처음에는 공개 자막 또는 자동 자막을 가져와서 요약하려고 했다. 하지만 실제로 돌려보면 업로드 직후에는 transcript가 아직 없거나, YouTube transcript API가 실패하는 경우가 있다.

그래서 처리 순서를 이렇게 잡았다.

```text
1. 공개 transcript 수집 시도
2. transcript가 있으면 transcript 기반 요약
3. transcript가 없으면 Gemini의 YouTube URL video input으로 요약
```

이 fallback이 중요하다. 아침 8시에 영상이 올라오고 8시 45분쯤 라이브가 끝나는 식이라면, 8시 50분에는 transcript가 아직 준비되지 않았을 수 있다. 이때 transcript만 바라보면 자동화가 자주 실패한다.

## AI 요약 프롬프트

요약은 뉴스 영상용으로 다음 기준을 줬다.

```text
다음 YouTube 뉴스 영상을 한국어로 요약해줘.

요구사항:
- 원문에 없는 사실을 추가하지 말 것
- 중요한 수치, 기업명, 정책, 시장 반응을 구체적으로 정리
- 투자 판단을 단정하지 말고 관찰 포인트로 표현
- 자동 자막 또는 영상 분석상 불확실한 표현은 별도로 표시

출력 형식:
## 3문장 요약
## 핵심 내용
- 5~8개 bullet
## 주요 수치와 고유명사
## 시장/비즈니스 관찰 포인트
## 확인이 필요한 부분
```

뉴스 요약에서는 “투자 추천처럼 보이지 않게” 하는 것이 중요하다. 그래서 `매수/매도` 같은 결론보다는 시장 관찰 포인트 중심으로 정리하게 했다.

## 운영 중 만난 문제

첫 번째 문제는 Notion 권한이었다.

GitHub Actions가 YouTube 영상은 찾았지만 Notion DB에 접근하지 못했다. 해결 방법은 Notion 데이터베이스를 integration에 공유하는 것이었다.

두 번째 문제는 timeout이었다.

외부 웹페이지나 YouTube oEmbed 응답이 늦어지면 workflow가 실패했다. 그래서 다음처럼 바꿨다.

- 개별 HTTP 요청 timeout을 짧게 설정
- 실패한 수집원은 건너뛰고 다음 대상으로 진행
- 로그에 현재 단계 출력
- 보조 페이지에서 확인하는 YouTube 링크 수 제한

자동화는 한 번 성공하는 것보다, 실패했을 때 어디서 왜 실패했는지 빨리 알 수 있게 만드는 것이 더 중요하다.

## 최종 결과

최종적으로 GitHub Actions에서 workflow가 성공했고, Notion DB에 YouTube 뉴스 요약 페이지가 생성되었다.

운영 구조는 다음과 같다.

```text
매일 08:50 KST
→ GitHub Actions 실행
→ 프로그램별 최신 영상 최대 2개 확인
→ Gemini로 요약
→ Notion에 원문 링크와 요약 저장
```

이 방식은 특정 뉴스 채널뿐 아니라 반복해서 보는 강의, 실적 발표, 세미나 영상에도 그대로 응용할 수 있다. 핵심은 수집 대상을 명확히 하고, 중복 저장 기준을 만들고, secret을 코드 밖에서 관리하는 것이다.
