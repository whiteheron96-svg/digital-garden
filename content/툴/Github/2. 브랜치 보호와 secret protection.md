
몇개의 리포지토리를 만들어 작업을 하다보니, 테스트용으로 만든 기능을 직접 메인 홈페이지에 배포하는 게 리스크가 크다고 느꼈다. 다른 브랜치를 활용해 테스트하는 것들을 공부하는 중이다. 

교보문고에서 관련 책을 찾다가, 눈에 띈 'github action으로 구현하는 실전 CI/CD' 책이 눈에 띄어서, 실전편 부분부터 읽고 있다. 개발자가 아니기에 기본적인 환경 설정, 리포지토리 깔끔하게 관리하는 방법, 버전 관리 체계 등에 대해 앞으로 많은 공부가 필요할 것 같다. 용어들도 아직 잘 많이 잘 모른다. 

보안취약점이 있으면 알림이 오거나, 외부 서비스 연결에 이슈상 확인해야 할 게 있으면 자동으로 탐지해주는 기능이 있다니 신기하다.

# GitHub 저장소 보안/품질 관리 용어 정리

  
GitHub의 보안/품질 설정은 팀 프로젝트에서만 필요한 것이 아니다. 혼자 개발하더라도 `main` 브랜치에 실수로 깨진 코드가 들어가거나, API 키가 노출되거나, 취약한 패키지를 오래 쓰는 문제를 줄일 수 있다.
  

특히 배포되는 프로젝트, 포트폴리오 프로젝트, 장기 유지보수 프로젝트라면 기본적인 보호 설정을 해두는 것이 좋다.
 

## 핵심 개념
### Branch protection rule

  중요한 브랜치, 보통 `main`이나 `develop`을 보호하는 규칙이다.

  직접 push를 막고, PR 리뷰나 CI 통과 후에만 merge되게 할 수 있다.

### Direct push

  브랜치에 바로 코드를 push하는 것.

빠르지만 리뷰, 테스트, 기록을 우회할 수 있어서 중요한 브랜치에서는 보통 막는다.

### Pull Request, PR

  내 작업 브랜치의 변경사항을 `main` 같은 브랜치에 합쳐 달라고 요청하는 공간이다.

  코드 변경 내용, 리뷰, 토론, CI 결과가 함께 기록된다.

### CI, Continuous Integration

  코드가 올라올 때마다 자동으로 테스트, 빌드, 린트 등을 실행하는 시스템이다.
 

GitHub에서는 보통 GitHub Actions로 구성한다.

  

### CODEOWNERS

  

특정 파일이나 폴더의 담당자를 지정하는 파일이다.

  

팀 프로젝트에서는 중요한 영역의 변경에 대해 자동으로 리뷰어를 지정할 수 있다.

  

### Dependency graph

  

프로젝트가 어떤 외부 패키지를 쓰는지 GitHub가 분석하는 기능이다.

  

Dependabot alerts와 security updates의 기반이 된다.

  

### Dependabot alerts

  

내 프로젝트가 사용하는 패키지에 보안 취약점이 있으면 알려주는 기능이다.

  

### Dependabot security updates

  

보안 취약점을 해결할 수 있는 패키지 버전이 있으면 Dependabot이 자동으로 PR을 만들어주는 기능이다.

  

### Grouped security updates

  

여러 보안 업데이트 PR을 가능한 한 묶어서 만들어주는 기능이다.

  

PR이 너무 많이 생기는 것을 줄여준다.

  

### Dependabot version updates

  

보안 문제가 없어도 새 버전이 나오면 업데이트 PR을 만들어주는 기능이다.

  

편리하지만 PR이 자주 생길 수 있다.

  

### Secret protection

  

API 키, 토큰, 비밀번호 같은 민감한 값이 저장소에 올라가는 것을 탐지하거나 push 단계에서 막는 기능이다.

  

### Secret scanning

  

이미 저장소에 올라간 커밋에서 API 키나 토큰 같은 시크릿을 탐지하는 기능이다.

  

### Push protection

  

push하려는 코드에 시크릿이 포함되어 있으면 GitHub가 막거나 경고하는 기능이다.

  

### Secrets and variables

  

GitHub Actions에서 사용할 비밀값을 저장하는 곳이다.

  

API 키를 코드에 직접 넣지 않고 `secrets.OPENAI_API_KEY` 같은 방식으로 사용하게 해준다.

  

## 혼자 개발할 때도 필요한가?

  

작은 연습용 프로젝트라면 꼭 필요하지 않다.

  

하지만 아래에 해당하면 설정하는 것이 좋다.

  

- 실제 배포되는 서비스

- 포트폴리오로 보여줄 프로젝트

- 장기간 유지보수할 프로젝트

- API 키나 DB 주소 같은 민감한 설정이 있는 프로젝트

- 테스트/빌드 자동화를 해두고 싶은 프로젝트

  

## 혼자 개발 기준 추천 설정

  

### Branch protection

  

```text

- main 직접 push 금지

- PR을 통해 merge

- CI 통과해야 merge 가능

```

  

### Advanced Security

  

```text

- Dependency graph 켜기

- Dependabot alerts 켜기

- Dependabot security updates 켜기

- Grouped security updates 켜기

```

  

### 처음에는 선택

  

```text

- Dependabot version updates

```

  

### 보통 필요 없음

  

```text

- Dependabot on self-hosted runners

```

  

### Secrets

  

```text

- API 키는 Secrets and variables -> Actions에 저장

- .env는 .gitignore에 추가

```

  

## 정리

  

GitHub의 보호 규칙과 보안 기능은 거창한 팀 개발 문화가 아니라, 실수를 줄이고 프로젝트를 안정적으로 유지하기 위한 기본 안전장치다.

  

혼자 개발하더라도 `main` 브랜치를 보호하고, PR과 CI를 거치며, Dependabot과 Secrets 관리를 설정해두면 나중에 발생할 수 있는 많은 문제를 미리 줄일 수 있다.
