# Supabase DB를 처음 백업해본 기록 (feat. 수많은 에러)

> Supabase로 운영하는 DB를 직접 백업해보고, 백업 파일을 안전하게 관리하고, 자동화 스크립트까지 만들어본 과정을 정리했다. 처음 해보는 거라 에러를 정말 많이 만났는데, 그 에러들이 오히려 공부가 됐다.

---

## 들어가며 — 왜 백업이 필요했나

Supabase로 서비스를 운영하다 보면 "데이터가 날아가면 어쩌지?" 하는 불안이 생긴다. 무료 플랜은 자동 백업을 제공하지 않기 때문에(Pro 플랜부터 7일치 자동 백업 제공), 무료로 쓰는 동안에는 직접 백업하는 방법을 알아둘 필요가 있었다.

먼저 짚고 넘어갈 개념. **Supabase는 PostgreSQL 기반 서비스다.** DB 본체는 PostgreSQL이고, 거기에 인증(Auth), 파일 저장(Storage), 자동 API 같은 편의 기능을 얹은 게 Supabase다. 그래서 백업도 PostgreSQL의 표준 도구인 `pg_dump`를 사용한다.

---

## 1단계 — 연결 정보 찾기

가장 먼저 한 일은 DB 연결 정보를 찾는 것이었다. 그런데 Supabase UI가 바뀌어서 예전 가이드에 나오는 "Settings → Database" 메뉴가 없었다. 한참 헤맸는데, 결국 이 경로로 찾았다.

대시보드 상단의 **Connect** 버튼을 누르면(또는 검색창에 "connect" 입력) **Connection string** 항목이 나온다. 여기서 Host, Port, Username, Database 이름을 확인할 수 있다.

---

## 2단계 — Direct Connection의 함정 (IPv6 문제)

처음엔 Connection string에 나온 Direct connection 주소로 백업을 시도했다. 그런데 계속 `unknown host`, `nodename nor servname provided` 같은 에러가 났다.

원인은 의외였다. **Direct connection은 IPv6 전용**인데, 내 네트워크(그리고 대부분의 가정/회사 네트워크)는 IPv4였던 것이다. Supabase 화면에도 작게 "Not IPv4 compatible"이라고 적혀 있었다.

해결책은 **Session Pooler**를 쓰는 것이었다. Pooler는 중간에 연결 관리자(PgBouncer)를 두는 방식인데, IPv4·IPv6를 모두 지원해서 백업에도 문제없이 동작한다.

||Direct Connection|Session Pooler|
|---|---|---|
|연결 방식|DB에 1:1 직접|중간 관리자 경유|
|IPv4 지원|❌ (IPv6 전용)|✅|
|백업 용도|IPv4에선 불가|✅|

여기서 중요한 디테일 하나. **Pooler를 쓸 때는 username 형식이 다르다.** 일반적으로 `postgres`가 아니라 `postgres.<프로젝트ID>` 형태여야 한다. 이걸 몰라서 `tenant/user ... not found` 에러를 또 만났다.

---

## 3단계 — 백업 실행, 그리고 조용한 성공

연결 정보를 제대로 맞추고 나니 다음 에러는 비밀번호였다. `password authentication failed`. 프로젝트 만들 때 설정한 DB 비밀번호를 정확히 입력해야 했다.

그렇게 모든 걸 맞춰서 실행했는데... 아무 반응이 없었다. 성공인지 멈춘 건지 알 수가 없었다. 알고 보니 **`pg_dump --format=custom`은 성공해도 아무 메시지를 출력하지 않는다.** 조용히 끝나는 게 정상이었던 것이다. (비밀번호 입력할 때 화면에 아무것도 안 보이는 것도 정상이다.)

성공 여부는 파일이 생겼는지로 확인했다.

```bash
ls -lh backup_*.dump
```

`297K` 크기의 `.dump` 파일이 보였다. 드디어 성공.

---

## 4단계 — 백업 내용 확인하기

백업한 파일이 뭘 담고 있는지 보고 싶었다. `.dump`는 압축 형식이라 그냥 열면 깨져 보인다. 방법은 몇 가지가 있었다.

서버 없이 내용 목록만 보려면 `pg_restore --list`, 사람이 읽을 수 있는 SQL로 변환하려면 `pg_restore -f`를 쓴다. 로컬 PostgreSQL에 복원해서 `psql`로 직접 쿼리를 쳐볼 수도 있는데, 이건 로컬 서버가 필요하다.

> 참고: 로컬 PostgreSQL 설치가 깨져 있으면 서버가 안 뜬다. `brew services list`로 상태를 보고, 로그(`/opt/homebrew/var/log/postgresql@<버전>.log`)를 확인하면 원인이 나온다. 설치가 불완전하면 완전히 지우고 재설치해야 한다. 단, 단순히 데이터 구경이 목적이라면 로컬 서버 없이 `pg_restore -f`로 SQL 변환만 해도 충분하다.

---

## 5단계 — 가장 중요한 것, 보안

백업 파일에는 **실제 데이터가 통째로 들어있다.** 회사 데이터라면 더더욱 조심해야 한다. 여기서 배운 게 가장 값졌다.

특히 회사 운영 DB를 개인 기기에 받는 것 자체가 보안 정책 위반일 수 있다는 점. 한국은 개인정보보호법도 있어서, 사용자 정보가 담긴 DB를 임의로 개인 노트북에 보관하면 문제가 될 수 있다. 그래서 실제 업무 데이터를 다룰 때는 **회사 정책을 먼저 확인**하고, 실습용 백업은 확인이 끝나면 삭제하는 게 안전하다.

파일을 보관해야 한다면 최소한 이 정도는 지켜야 한다. Git에 절대 올리지 않기(`.gitignore` 등록), 파일 권한을 나만 읽도록 제한하기(`chmod 600`), 클라우드 동기화 폴더(Google Drive, Dropbox 등)는 피하기, 필요 없어지면 완전히 삭제하기.

---

## 6단계 — 비밀번호를 코드에 박지 않기 (.env 분리)

매번 비밀번호를 입력하기 번거로워서 백업을 스크립트로 만들었다. 핵심은 **비밀번호 같은 민감 정보를 `.env` 파일에 따로 빼고, 스크립트에서는 변수로만 참조**하는 것이다. 이렇게 하면 코드에 비밀번호가 박히지 않는다.

`.env` 파일 (민감 정보만 모음):

```bash
PGPASSWORD=실제비밀번호
SUPABASE_HOST=OOO.supabase.com
SUPABASE_USER=postgres.프로젝트ID
```

`backup.sh` 파일 (`.env`를 읽어서 실행):

```bash
#!/bin/bash
cd "$(dirname "$0")"   # 스크립트가 있는 폴더로 이동
source .env            # 같은 폴더의 .env를 읽어 변수로 불러옴

mkdir -p $HOME/db_backups   # 폴더 없으면 생성 (있으면 그냥 넘어감)

pg_dump \
  --host=$SUPABASE_HOST \
  --port=5432 \
  --username=$SUPABASE_USER \
  --dbname=postgres \
  --format=custom \
  --file=$HOME/db_backups/backup_$(date +%Y%m%d).dump

echo "백업 완료!"
```

여기서 알게 된 것들:

- `source .env`는 **현재 작업 디렉토리**에서 `.env`를 찾는다. 그래서 `cd "$(dirname "$0")"`를 넣어 스크립트 위치 기준으로 맞췄다.
- `$HOME`은 직접 입력하는 값이 아니라 시스템이 자동으로 아는 변수다(`/Users/사용자이름`). `.env`에 따로 적을 필요 없다.
- `pg_dump`는 폴더를 자동 생성하지 않는다. 저장 폴더가 없으면 백업이 아예 안 되므로 `mkdir -p`로 미리 챙겨야 한다. `-p` 옵션은 폴더가 있으면 넘어가고 없으면 만들어줘서 매번 실행해도 안전하다.

---

## 7단계 — Git에 올리기 (.env는 빼고)

스크립트와 가이드 문서를 GitHub에 올리되, `.env`만 제외했다. **순서가 중요하다.** 한 번 git에 올라간 파일은 나중에 지워도 기록에 남기 때문에, `.gitignore`를 먼저 등록한 다음 커밋해야 한다.

먼저 `.gitignore`를 만들고:

```bash
echo ".env" >> .gitignore
echo "*.dump" >> .gitignore
```

`git add .` 후 `git status`로 **올라갈 목록에 `.env`가 없는지** 반드시 눈으로 확인했다. `.gitignore`가 제대로 동작하면 초록색 목록에서 `.env`가 사라진다. 확인 후 커밋·푸시했고, 마지막으로 GitHub 웹사이트에서 `.env`가 안 보이는지 직접 확인하는 것으로 마무리했다.

> 만약 `git status`에서 `.env`가 이미 추적 중(목록에 보임)이라면, `git rm --cached .env`로 추적에서 빼야 한다. `--cached`는 git 기록에서만 빼고 실제 파일은 지우지 않는다.

---

## 마치며

처음 해보는 백업이라 호스트 주소, IPv4/Pooler, username 형식, 비밀번호, 로컬 서버 설치까지 거의 모든 단계에서 막혔다. 하지만 에러 하나하나가 "왜 이렇게 동작하는가"를 이해하게 해줬다. 특히 **민감 정보를 코드에서 분리하고 git에서 차단하는 흐름**은 앞으로 어떤 프로젝트에서든 그대로 쓸 수 있는 습관이 됐다.

다음 목표는 GitHub Actions로 백업을 자동화하는 것. 다만 회사 데이터의 보관 위치와 권한은 회사 정책에 맞춰야 하므로, 그건 확인 후에 진행할 예정이다.

# 🔐 꼭 지켜야 할 보안 수칙 (가장 중요)

### 1. 비밀번호·키를 코드에 직접 쓰지 않는다

비밀번호, API 키, 토큰은 `.env` 같은 별도 파일에 분리하고, 코드에서는 변수로만 참조한다. 코드에 직접 박으면(하드코딩) 그 코드가 공유되는 순간 함께 유출된다.

### 2. `.env`와 백업 파일은 절대 Git에 올리지 않는다

`.gitignore`에 `.env`, `*.dump`를 **반드시 먼저** 등록한다. 한 번 커밋된 파일은 나중에 지워도 git 기록에 영구히 남으므로, 올리기 전에 막는 게 핵심이다.

### 3. 커밋 전 `git status`로 눈으로 확인한다

`git add .` 후 `git status`를 쳐서, 올라갈 목록(초록색)에 민감 파일이 없는지 매번 확인한다. 푸시 후에는 GitHub 웹에서 한 번 더 확인한다.

### 4. 실수로 올렸다면 추적에서 빼되, 비밀번호는 교체한다

`git rm --cached 파일명`으로 추적에서 뺀다. 하지만 이미 푸시까지 됐다면 기록에 남아있을 수 있으니, **노출된 비밀번호·키는 반드시 새로 발급/변경**한다.

### 5. 백업 파일은 실제 데이터 덩어리임을 잊지 않는다

`.dump` 파일에는 데이터가 통째로 들어있다. 파일 권한을 제한하고(`chmod 600`), 클라우드 동기화 폴더(Google Drive, Dropbox, NAS)는 피하며, 필요 없어지면 완전히 삭제한다.

### 6. 회사 데이터는 임의로 다루지 않는다

회사 운영 DB를 개인 기기에 받거나 보관하는 것은 보안 정책 위반일 수 있다. 보관 위치·암호화·접근 권한은 **회사 정책을 먼저 확인**하고 따른다. 실습용이면 확인 후 삭제가 가장 안전하다.

### 7. 비밀번호 입력이 화면에 안 보이는 건 정상이다

터미널에서 비밀번호를 칠 때 글자나 점이 안 보이는 건 보안을 위한 정상 동작이다. 그대로 입력하고 Enter를 누르면 된다.
